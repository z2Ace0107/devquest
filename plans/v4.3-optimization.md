### Bug 5: services.py usage_count 语义扭曲 ✅ 已确认

- **文件**: `backend/services.py:176-178`
- **现象**: `helpful=True → usage_count += 10`，`helpful=False → usage_count -= 2`（不低于 0）
- **影响**: usage_count 字段名是"使用次数"，但实际变成了"反馈加权分"。5 次有用 = 50，与真实使用次数严重脱节，且 RRF boost 基于 `min(usage_count, 10) * 0.03`，导致 helpful 反馈后直接满额 30% boost
- **修复**: 改为 +1/-1 保持语义一致，并调整 RRF boost 公式

```python
# services.py 修复:
if helpful:
    problem.usage_count = old_usage + 1
else:
    problem.usage_count = max(0, old_usage - 1)

# vector_search.py _rrf_fusion 调整 boost 公式:
# 旧: boost = 1.0 + min(usage, 10) * 0.03   → 最高 +30%
# 新: boost = 1.0 + min(usage, 5) * 0.05     → 最高 +25%（5次有用）
# 这样 5 次有用反馈 = 25% boost，更合理
```

### Bug 11: _probe_llm() 缓存 ✅ 已确认

- **文件**: `backend/llm_client.py:204-207`
- **修复**: 加 5 分钟缓存，和 fallback cooldown 对称

```python
_probe_cache: dict[str, tuple[bool, float]] = {}  # provider → (ok, timestamp)
_PROBE_CACHE_TTL = 300  # 5 分钟

def _is_probe_valid(name: str) -> bool:
    """检查最近一次探测是否仍在有效期内。"""
    ok, ts = _probe_cache.get(name, (False, 0))
    return ok and (time.time() - ts) < _PROBE_CACHE_TTL

# get_llm() 中探测逻辑改为:
for name, cfg in [("primary", PRIMARY), ("fallback", FALLBACK)]:
    if _is_probe_valid(name):
        # 缓存命中，跳过探测
        llm = _get_or_create(name, cfg, temperature)
        return llm
    ...
    try:
        llm = _get_or_create(name, cfg, temperature)
        _probe_llm(llm)
        _probe_cache[name] = (True, time.time())  # 探测成功，缓存
        ...
    except Exception:
        _probe_cache[name] = (False, time.time())  # 探测失败，也缓存
```
